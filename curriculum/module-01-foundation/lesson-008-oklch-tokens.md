## Context

The single biggest predictor of whether a design system survives three redesigns is whether it had a token layer on day one. Tokens decouple intent ("primary brand color") from value (`oklch(66% 0.19 55)`). When a component references `var(--color-brand)` instead of `#ff6b35`, you can swap the brand without touching a single component. When you build a dark theme, you override the alias layer, not every component's CSS.

We establish the token layer now, before any component exists. The token file is structured in two tiers: **primitives** (raw color ramps like `--color-neutral-500`, `--color-primary-600`) and **aliases** (semantic names like `--color-bg`, `--color-fg`, `--color-brand`). Components reference aliases almost exclusively. The primitive layer is for one-off needs inside the token file itself — when an alias needs a specific step of a ramp.

Every color value is expressed in **OKLCH**, not hex or HSL. Why OKLCH matters — and why other courses skip this step — is the substance of this lesson.

## The command

Create the tokens file:

```bash
mkdir -p src/lib/styles
touch src/lib/styles/tokens.css
```

Write `src/lib/styles/tokens.css` with the full token system. Structure: `@layer tokens { :root { /* primitives then aliases */ } }` — wrapping in a named cascade layer now means lesson 011's full `@layer` cascade declaration can order this layer deterministically against reset, base, components, utilities, and overrides.

Primitive scales we define (each is 11 steps: 50 → 950):
- **neutral** — warm-tinted gray (hue 60°), very low chroma
- **primary** — forge warm amber (hue 55°), moderate chroma at 500
- **accent** — electric blue (hue 250°), moderate chroma at 500

Semantic scales (4 stops each: 50, 500, 700, 900):
- **success** (hue 145°, green)
- **warning** (hue 80°, amber)
- **danger** (hue 25°, red)
- **info** (hue 220°, blue)

Alias layer:
- `--color-bg`, `--color-bg-raised`, `--color-bg-sunken`, `--color-bg-inverse`
- `--color-fg`, `--color-fg-muted`, `--color-fg-subtle`, `--color-fg-inverse`
- `--color-border`, `--color-border-strong`, `--color-border-inverse`
- `--color-brand`, `--color-brand-hover`, `--color-brand-fg`
- `--color-link`, `--color-link-hover`
- `--color-focus-ring`

Non-color primitives also live in this file: `--radius-*`, `--shadow-*`, `--duration-*`, `--easing-*`. They earn their seat here because they are design tokens — values that the whole system references and that we want changeable in one place.

Dark mode is handled two ways:

1. `@media (prefers-color-scheme: dark)` — respects the OS preference out of the box
2. `:root[data-theme='dark']` — explicit override for a future user-controlled theme toggle

Both blocks redefine only the **alias** layer. The primitive ramps are invariant across themes.

Wire the tokens into the app via the root layout:

```diff
 <script lang="ts">
 	import favicon from '$lib/assets/favicon.svg';
+	import '$lib/styles/tokens.css';

 	let { children } = $props();
 </script>
```

Importing CSS from a `.svelte` script block is a Vite-idiomatic pattern: Vite extracts the CSS into a stylesheet link and hoists it into the document head. The import runs once at app boot because the root `+layout.svelte` wraps every page. Every component, everywhere, has `var(--color-brand)` available from the first paint.

Verify everything compiles:

```bash
pnpm check
pnpm build
```

Expected: 0 errors, clean build.

## Why we chose this — the PE7 judgment

**Alternative 1: HSL instead of OKLCH**
HSL looks similar in syntax — three numbers — but is perceptually uneven. Two HSL colors with identical lightness values (`hsl(200 50% 50%)` and `hsl(60 50% 50%)`) look dramatically different in brightness because HSL's lightness axis corresponds to arithmetic average, not perceived lightness. The practical consequence: when you build a 10-step scale in HSL by linearly interpolating lightness, the mid-tones look wrong — blues and greens appear darker than reds and yellows at the same "50%" step. You end up hand-tuning every step, and when you add a new color family you hand-tune again. In OKLCH, 50% lightness is 50% perceived lightness across every hue. A scale is generated by varying one number; it just works.

**Alternative 2: Hex codes (`#ff6b35`)**
Hex is the lowest common denominator. It says nothing about the color: you cannot tell from `#ff6b35` whether it's 60% lightness or 40%, whether it's saturated or desaturated, what hue family it belongs to. Transforming a palette — making the brand 10% darker, rotating to a colder hue — requires a color picker and manual re-entry. In OKLCH, changing `oklch(66% 0.19 55)` to `oklch(60% 0.19 55)` is a straightforward 6% lightness reduction. The language of the color value matches the language of the design intent.

**Alternative 3: A design-token generator (Style Dictionary, Tokens Studio, Figma-export pipeline)**
Generators produce JSON → CSS via a build step. They're excellent at scale — a design system consumed across 12 apps on 4 platforms (iOS, Android, web, Figma). For a single-app codebase, they add a build step, a config file, and a pipeline dependency in exchange for... writing CSS variables. We are writing CSS variables. A plain `.css` file edited by hand is fewer moving parts and produces identical output. When ForgeSchool grows into a multi-app platform, we can introduce a generator and migrate the tokens; the CSS custom property API is unchanged. The generator is a premature optimization in 2026.

**Alternative 4: CSS preprocessor variables (Sass, Less)**
Preprocessor variables are compile-time — they don't exist in the browser. A dark-mode override means re-compiling for each theme, serving two stylesheets, or using a runtime JS shim. CSS custom properties are runtime — `color: var(--color-bg)` reads the current computed value, which respects cascading overrides from `@media (prefers-color-scheme)` or `[data-theme]`. For theming, custom properties are strictly more capable. Sass is excellent for math and mixins, but for the token story, `:root { --... }` wins.

**Alternative 5: Tailwind's color palette via `@apply`**
Tailwind bakes palettes at config-file authoring time and exposes them as utility classes. Two consequences. First, every component's class list becomes part of the token surface — if you want to change the brand from blue to amber, you grep and replace `bg-blue-500` → `bg-amber-500` across the entire codebase. Second, CSS custom properties still win for runtime theming; Tailwind's dark mode uses `[data-theme='dark']` selectors to override each utility, which is a less elegant mechanism than overriding a single `--color-bg` alias. Tailwind is a styling convention; token-first CSS is a design-system convention. They are not equivalent, and for PE7 we want the design system.

The PE7 choice — OKLCH primitives + semantic aliases in a single hand-maintained `tokens.css` — wins because OKLCH is perceptually correct, aliases enable theming without touching components, and the system has zero build-time dependencies.

## What could go wrong

**Symptom:** Colors look washed out or muted compared to a Figma design
**Cause:** Chroma values in OKLCH are absolute and the upper bound varies by hue. A chroma of `0.2` at hue 250° (blue) is inside sRGB gamut; at hue 30° (orange) it might be out of gamut and the browser will clamp to display. If a design system mockup uses pure sRGB extremes, OKLCH cannot match them faithfully on an sRGB display — but it will match them faithfully on a P3 display.
**Fix:** Check each color in a tool like [oklch.com](https://oklch.com). If a color shows as out-of-gamut for sRGB, lower the chroma by 0.02 increments until it fits. For OKLCH values intended for wide-gamut displays, specify a fallback via `@supports not (color: oklch(0 0 0))` — modern Safari, Chrome, and Firefox all support OKLCH without fallback, so this is needed only for very old browser support.

**Symptom:** Dark mode doesn't apply on macOS with OS set to dark
**Cause:** The `color-scheme: light` declaration inside `:root` overrides the `@media (prefers-color-scheme: dark)` block because both have equal specificity and the light one comes second in source order... actually no, that's not right either. The real cause is usually that the tokens file isn't loaded before paint, so the initial render uses browser defaults.
**Fix:** Make sure the import is in `+layout.svelte`, not in a child route — the root layout is rendered first. Also confirm the `@media` block is outside the `:root` rule, not nested inside it (nested-at-rules are a newer spec feature with uneven support).

**Symptom:** A component references `var(--color-bg-primary)` and gets `inherit` or nothing
**Cause:** Typo'd alias name. There is no `--color-bg-primary` in this token file — it's `--color-bg` (the page background) and `--color-brand` (the primary brand color). Two different things.
**Fix:** Read the alias section of `tokens.css` as the canonical list. If a new alias is needed, add it to the file rather than inventing one in a component.

**Symptom:** Opening the production build shows unstyled flash of text for half a second
**Cause:** CSS is being loaded async rather than inlined. The token file is small enough (~5KB) that SvelteKit should inline it, but if the build output shows a separate `.css` link it may be getting fetched after HTML parse.
**Fix:** Inspect the build output. SvelteKit's default behavior is to inline small critical CSS in the head. If you see a flash, measure with Lighthouse and check whether `preload` hints are missing.

## Mistake log — things that went wrong the first time I did this

- **Tried to generate the primitive ramp by scripting OKLCH with a linear lightness step.** The first attempt had `--color-neutral-500` at 55% lightness and `--color-neutral-900` at 15%, which looked great in isolation but produced insufficient contrast at body-text sizes against `--color-neutral-50` backgrounds. Adjusted so that 700/800/900 are darker than a naive linear ramp would suggest (34/24/16), giving text weights real visual weight. The takeaway: test contrast with real text at small sizes before committing to a ramp.
- **Left `color-scheme` undeclared.** Without `color-scheme: light` / `color-scheme: dark` on `:root`, native form controls and scroll bars use the browser's default chrome, which doesn't match the theme. Added the declaration, and suddenly `<input>` borders, scroll bars, and caret colors all respected the theme.
- **Wrapped only the primitives in `@layer tokens` and left aliases at root scope.** The first build order had primitives layered and aliases unlayered, which meant later `@layer utilities` rules could not override alias values without `!important`. Wrapped everything in a single `@layer tokens` block — the whole token system is now one layer in the cascade, and overrides happen cleanly via later layers.
- **Used transparent white for shadows instead of OKLCH zeros.** Shadows specified as `rgba(0, 0, 0, 0.1)` work but mix inconsistently across themes. `oklch(0% 0 0 / 0.1)` is OKLCH's way of saying "pure black at 10% opacity" and composes more predictably in P3 color space. Switched all shadow declarations.
